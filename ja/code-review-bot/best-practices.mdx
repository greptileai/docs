このガイドでは、実際のチームの経験に基づき、Greptile から最大の価値を引き出すための実証済みの戦略を紹介します。機能の概要は [開発者向けクイックリファレンス](/ja/developer-quick-reference) を参照してください。

## 把握しておきたい設定オプション

まだ Greptile をセットアップしていない場合は、[はじめにガイド](/ja/code-review-bot/getting-started)から始めてください。

### ダッシュボードの設定

利用可能な設定の詳細は、[主要機能](/ja/code-review-bot/key-features)ガイドをご覧ください。

#### Greptile のコメント数を調整する

重大度しきい値は、Greptile のコメント数をコントロールするための主要な手段です。

<CardGroup cols={3}>
  <Card title="Low Threshold">
    **コメント多め、詳細多め**

    - 潜在的な問題にもコメント
    - 軽微な提案も含む
    - 詳細なフィードバックを重視するチームに最適
  </Card>

  <Card title="Medium Threshold">
    **バランス重視（デフォルト）**

    - 軽微な問題と重大な問題のバランス
    - 細かすぎる指摘を多く除外
    - 最初の設定として推奨
  </Card>

  <Card title="High Threshold">
    **重大な問題のみ**

    - 少数だが影響度の高いコメント
    - バグやセキュリティに注力
    - ボットのノイズが多いと感じるチームに最適
  </Card>
</CardGroup>

<Tip>
  まずは中程度のしきい値から始め、チームのフィードバックに応じて調整してください。ノイズが多すぎる場合は上げ、より詳細が欲しい場合は下げましょう。
</Tip>

#### Greptile のコメント種別を調整する

Greptile がコメントする問題の種類を制御できます。

- **Syntax**: 基本的なコード構造やフォーマット
- **Logic**: 潜在的なバグや論理的な問題
- **Style**: コーディング規約やベストプラクティス

<Note>
  多くのチームは既にリンターを使っている場合、構文に関するコメントを無効にし、Greptile が自動化ツールでは見逃しやすいロジックやスタイルの問題に集中できるようにしています。
</Note>

## Greptile にあなたのコードベースとパターンを学習させる

Greptile の学習方法については、[学習／メモリ](/ja/how-greptile-works/memory-and-learning) ガイドをご覧ください。

### 効果的なコメント対応

<Steps>
  <Step title="コメントに反応する">
    役立つコメントには 👍、役立たないコメントには 👎 を付けます。これは Greptile があなたの好みを学習する主要な手段です。
  </Step>

  <Step title="フィードバックを説明する">
    低評価する際は、理由を添えて返信してください: `@greptileai 実は、可読性が高いのでリスト内包表記を好みます`
  </Step>

  <Step title="質問する">
    明確化が必要な場合は `@greptileai` をメンションしてください: `@greptileai なぜこれがセキュリティ上の問題なのか説明してもらえますか？`
  </Step>

  <Step title="一貫性を保つ">
    チーム全体で定期的かつ一貫したフィードバックを行うと、Greptile の学習が加速します。
  </Step>
</Steps>

### 学習タイムライン

<Info>
  一貫したフィードバックを**2〜3週間**続けると、コメントの関連性と正確性に目に見える改善が得られるのが一般的です。
</Info>

最初の数週間は、Greptile がチームの嗜好を学習中のためノイズが多くなる場合があります。システムは次の点で向上します:

- チームが重視する提案の選別
- コードベースのコンテキストの理解
- チーム固有のコーディングパターンの認識

### Greptile は何から学習するのか？

- コメントに対するあなたの反応
- PR 上の他の開発者のコメント
- 実装された提案と無視された提案の傾向
- コードパターンに関するチームでの議論

### コンテキストファイルのベストプラクティス

既存のドキュメントを Greptile に参照させましょう:

- **スタイルガイド**: 言語ごとのコーディング標準
- **アーキテクチャドキュメント**: システム設計とパターン
- **セキュリティポリシー**: コンプライアンスとセキュリティ要件
- **API 標準**: 一貫した API 設計パターン
- **テスト戦略**: チームのテスト方針・進め方

<Warning>
  学習されたルールは定期的に見直してください。Greptile はチームのコメントに基づいて自動でルールを作成しますが、実際の方針と整合しているか確認する必要があります。
</Warning>

## 高度な設定

### リポジトリレベルの設定

リポジトリのルートにある `greptile.json` を使用して、リポジトリ単位の設定を行います。詳しくは [greptile.json ガイド](/ja/code-review-bot/greptile-json)をご覧ください:

```json
{
  "strictness": "medium",
  "commentTypes": ["logic", "style"],
  "patternsRepo": ["org/frontend-repo"],
  "customInstructions": "Focus on security and performance",
  "scope": {
    "include": ["src/**"],
    "exclude": ["tests/**", "docs/**"]
  }
}
```

### 分析モニタリング

ダッシュボードで次の主要指標を追跡します:

- **マージまでの平均時間**: レビューはマージを促進していますか、それとも遅らせていますか？
- **コメントのエンゲージメント**: 高い賛成率は有用なフィードバックの指標です
- **MRあたりの対応済みコメント数**: チームでの活用状況と価値を示します

### 連携の最適化

<AccordionGroup>
  <Accordion title="トリガー設定">
    次に基づいてスマートトリガーを設定します:

    - 特定の作成者（新規チームメンバー）
    - ブランチパターン（feature ブランチ）
    - PR ラベル（needs-review）
    - ファイルタイプ（クリティカルなシステムファイル）
  </Accordion>

  <Accordion title="パターンリポジトリ">
    より良いコンテキストのために関連リポジトリをリンクします:

    - バックエンドのレビューにフロントエンドのリポジトリを追加
    - 共有ライブラリのリポジトリを含める
    - マイクロサービスの依存関係を相互参照
  </Accordion>
</AccordionGroup>

## 回避すべき一般的な落とし穴

<Warning>
  **学習期間を軽視しないでください。** Greptile がチームの嗜好を把握するには、2〜3週間にわたる継続的なフィードバックが必要です。
</Warning>

<Warning>
  **初期段階で重大度を低く設定しないでください。** まずは保守的に始め、チームがフィードバック品質に慣れてきたらしきい値を下げてください。
</Warning>

<Warning>
  **カスタムルールの適用範囲を忘れないでください。** どこにでも適用される広範なルールは、不適切なコンテキストでノイズを生む原因になります。
</Warning>

<Info>
  覚えておいてください。Greptile は継続的に学習します。定期的な活用とフィードバックにより、時間とともに価値が高まります。
</Info>

## トラブルシューティングとFAQ

### よくある問題

<AccordionGroup>
  <Accordion title="Greptile が PR をレビューしてくれない">
    **考えられる原因:**

    - PR が設定済みのトリガールール（ラベル、作成者、ブランチ）に一致していない
    - 変更が除外対象のファイルまたはディレクトリに含まれている
    - リポジトリのインデックス化がまだ進行中（ダッシュボードで「updating」タグを確認）

    **解決策:**

    - 手動トリガーを試す：PR に `@greptileai` とコメントする（[手動トリガー](/ja/code-review-bot/trigger-code-review)を参照）
    - ダッシュボードの設定でチームのトリガー構成を確認
    - リポジトリが正しく接続され、インデックス化されているか確認
  </Accordion>

  <Accordion title="無関係なコメントが多すぎる">
    **解決策:**

    - ダッシュボード設定で重大度しきい値を上げる（Low → Medium → High）
    - 不要なコメント種別を無効化する（リンターを使用している場合は構文など）
    - 無関係なコメントに継続的に低評価を付けて Greptile の学習を支援する
    - チームの方針を明確にするカスタムルールを追加する（[カスタムコンテキスト](/ja/code-review-bot/custom-context)を参照）
  </Accordion>

  <Accordion title="コメントが役に立たない／正確でない">
    **最初の 2〜3 週間は通常の挙動です。** Greptile は好みを学習するためにフィードバックを必要とします。

    **解決策:**

    - 役に立つコメントには一貫して高評価を、役に立たないコメントには低評価を付ける
    - なぜ誤っているのかを返信で説明する：`@greptileai 実は、このパターンは当社のコードベースで推奨されています。理由は...`
    - チームの標準やアーキテクチャ上の判断に関するカスタムコンテキストを追加する（[カスタムコンテキスト](/ja/code-review-bot/custom-context)を参照）
    - 焦らずに運用する—学習には時間がかかりますが、着実に改善します
  </Accordion>

  <Accordion title="リポジトリのインデックス化に時間がかかりすぎる">
    **大規模なコードベースでは通常のことです。** 初回のインデックス化は計算負荷が高く、数時間かかる場合があります。

    **想定されること:**

    - 大規模リポジトリ（10万行以上）：初回インデックスに数時間
    - その後の更新：はるかに高速で、新しい変更のみを処理
    - インデックス化中はダッシュボードに「updating」タグが表示される

    **停止している場合:** リポジトリの詳細とともにサポートに連絡してください
  </Accordion>

  <Accordion title="最近の変更後に Greptile が動作しなくなった">
    **考えられる一般的な原因:**

    - リポジトリの権限またはアクセストークンの有効期限が切れている
    - トリガールールが更新され、PR が除外されるようになっている
    - リポジトリが誤って切断されている

    **解決策:**

    - ダッシュボードでリポジトリの接続状態を確認
    - トリガールールがワークフローに合致しているか検証
    - 手動トリガーで Greptile が反応するかテストする（[手動トリガー](/ja/code-review-bot/trigger-code-review)を参照）
  </Accordion>
</AccordionGroup>

### よくある質問

<AccordionGroup>
  <Accordion title="Greptile がチームの好みを学習するまでどれくらいかかりますか？">
    複数のチームメンバーから継続的にフィードバックがある場合、**2〜3週間**です。次のような変化が見られます:

    - 1週目: Greptile が基本的なパターンを学習中のためノイズが多い
    - 2〜3週目: コメントの関連性が大幅に改善
    - 2か月目以降: チーム固有の好みに高精度にチューニング
  </Accordion>

  <Accordion title="古い PR やクローズ済みの PR でも Greptile を使えますか？">
    **はい。** `@greptileai` とコメントすれば、任意の PR でレビューを手動実行できます（[手動トリガー](/ja/code-review-bot/trigger-code-review)を参照）。次のケースで機能します:

    - Greptile 有効化前の古い PR
    - クローズ済みまたはマージ済みの PR（学習目的）
    - 通常は自動レビューをトリガーしない Draft PR
  </Accordion>

  <Accordion title="大きな PR や複雑な変更を Greptile はどう扱いますか？">
    Greptile は変更セット全体を解析し、以下を提供します:

    - 全体的な変更内容の自然言語による要約
    - 複雑な変更に対するシーケンス図
    - 大規模でリスクの高い変更に対する低めの信頼スコア
    - ファイル間の依存関係を把握したコンテキスト認識のコメント
  </Accordion>

  <Accordion title="Greptile のコメントに同意できない場合はどうすればよいですか？">
    **やり取りしてください:**

    - コメントに低評価を付けて Greptile に学習させる
    - 理由を説明して返信する: `@greptileai このパターンが好ましい理由は…`
    - 代替案を求める: `@greptileai 別のやり方はありますか？`
    - Greptile はフィードバックから学習し、以後のコメントを調整します
  </Accordion>

  <Accordion title="リポジトリごとに異なる設定にできますか？">
    **はい。** 複数の方法があります:

    - ダッシュボード設定はチーム内のすべてのリポジトリに適用
    - リポジトリのルートにある `greptile.json` でダッシュボード設定を上書き可能（[greptile.json ガイド](/ja/code-review-bot/greptile-json)参照）
    - カスタムコンテキストは特定のリポジトリやファイル種別にスコープ可能（[カスタムコンテキスト](/ja/code-review-bot/custom-context)参照）
    - 組織内のチームごとにまったく異なる構成にできる
  </Accordion>

  <Accordion title="カスタムコンテキストが使われているかどうかはどう確認できますか？">
    カスタムコンテキストのダッシュボードを確認してください（[カスタムコンテキスト](/ja/code-review-bot/custom-context)の詳細）:

    - 「Last Applied」列に各コンテキスト項目が使用された日時が表示されます
    - 任意のコンテキスト項目をクリックすると使用状況の統計が表示されます
    - 「Recent Usage」セクションにコンテキストが適用された具体的なコメントが表示されます
  </Accordion>

  <Accordion title="Greptile がセキュリティ問題を示唆した場合はどうすべきですか？">
    **真剣に受け止めてください:**

    - 同意できない場合でも、提案を慎重に検討する
    - 具体的な説明を求める: `@greptileai ここでのセキュリティリスクを説明してもらえますか？`
    - 潜在的な脆弱性を独自に調査する
    - 誤検知であれば、低評価して理由を説明し、Greptile の学習に役立てる
  </Accordion>
</AccordionGroup>
